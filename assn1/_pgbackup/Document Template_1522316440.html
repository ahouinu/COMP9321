Overview

This documentation covers the Coffee Shop web API for COMP9321 Assignment 1.

Media Type support

All server response bodies MUST be valid JSON Media Type messages.

All server request bodies MUST be valid query parameters.

Responses

The API returns three possible payloads (as JSON object arrays):

  1. Order Payload

  2. Payment Payload

  3. Error Payload

See below for details.

Order Payload

A Order object looks like this:

Cashier's Order Object
{
"cost": 5,
"datetime": "2018-03-28 23:49",
"extra":[
"Iced"
],
"id": 0,
"payment_link": "/payments/0",
"size": "Large",
"type": "Latte"
}
Barista's Order Object
{
"additions":[
"Iced"
],
"check_payment": "/payments/0",
"createdtime": "2018-03-28 23:49",
"id": 0,
"size": "Large",
"status": "Making Drink",
"type": "Latte"
}

The valid list of properties for a Orders object for Cashiers are: [id], [type], [size], [cost], [additions], [datetime], and [payment_link].

The valid list of properties for a Orders object for Baristas are: [id], [type], [size], [cost], [additions], [status], [datetime], and [check_payment].

See Data Elements for more information on each property.

Payments Payload

A Payment object looks like this:

Cashier's Payments Object
{
"card_details":[
"12345"
],
"datetime": "2018-03-28 23:56",
"id": 0,
"type": "card",
"amount": 5
}
Barista's Payments Object
{
"datetime": "2018-03-28 23:56",
"id": 0,
"amount": 5,
"paid": true
}

The valid list of properties for a Payment object for Cashiers are: [id], [type], [amount], [card_details], and [datetime].

The valid list of properties for a Payment object for Baristas are: [id], [type], [amount], [card_details], [paid]and [datetime].

See Data Elements for more information on each property.

Error Payload

If resource isn't in database:

Error Payload
code: 404
{
"message": "Not Found"
}

If order is not available:

Error Payload
code: 400
{
"message": "Bad Order"
}

If cashier try to update/cancel an order after payment:

Error Payload
code: 400
{
"message": "Already Paid"
}

Requests

All payloads sent from the client to the server MUST be valid query parameters. The messages SHOULD be in simple '?name=value' format based on the INPUT arguments from the Order Operations and the Payment Operations tables.

For example, the Order Operations table lists four arguments for the create_order() operation: [type], [size], and [addtions], and [checkPass]. When constructing a request body for this operation, it SHOULD look like this:

Create Order
/orders?type=Latte&size=Large&additions=Iced

Data Elements

What follows is a list of all the possible data elements that MAY appear within a coffee web API payload. Note that these values can appear in both requests and responses.

create_order()

Create a new order. This value of this data element MUST match an existing [type] [size] value. See Order Payload.

checkPass

Check value to use when changing the [oldPass] to [newPass]. The value of [checkPass] MUST exactly match the value of [newPass].

order_type

Coffee type. [LongBlack, Latte, FlatWhite, Espresso, Mocha, Macchiato]

payment_type

Payment type. [card, cash]

size

Coffee size. [Large, Small]

cost

Coffee cost, which is retrieved from the price_dict: 

price_dict = {'LongBlack': 4.00,
              'Latte': 4.50,
              'FlatWhite': 4.50,
              'Espresso': 3.50,
              'Mocha': 5.00,
              'Macchiato': 4.00}

additions

Extra requirements which will be stored in a list

datetime

System generated datetime in format%Y-%m-%d %H:%M

payment_link

System generated link to corresponding payment url/payments/{id}

status

Status string, which is retrieved from status_dict: 

status_dict = {-1: 'Od Cancelled',
              0: 'Od Received',
              1: 'Od Placed, Processing Payment',
              2: 'Paid',
              3: 'Making Drink',
              4: 'Drink Made',  # Not used
              5: 'Drink Served'}

amount

The amount that customer pays for his/her drink. If amount != cost, return 400 with message "Bad Order". See Error Payload

card_details

Card details for payments which are paid by card. Can be null if the customer pays by cash.

paid

Indicates whether a payment is finished or not.

Extensions

This document describes the Tasks CRUD service. Any extensions to this service MUST not redefine or change the use/meaning of any URLs, objects (or their properties), arrays, properties, etc. defined in this document. Clients that do not recognize extensions to the service SHOULD ignore them.

Warning
 It is possible that future forward-compatible modifications to this specification will include new elements, attributes, attribute values, and data types. Extension designers should take care to prevent future modifications from breaking or redefining those extensions.

The details of designing and implementing compatible extensions is beyond the scope of this document.
